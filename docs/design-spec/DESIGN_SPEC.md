# Bloom & Grace 作業版 Design Spec

## Background
- 本專案是六角 AI 開發進化營第一場作業延伸版本。
- 目前專案已具備花店電商前端、會員流程示意、購物車、本地 server 與綠界金流串接雛形。
- 但目前整體呈現偏向 AI 展示型網站，資訊量偏多、敘事偏滿，與作業驗收導向的專案感不夠一致。
- 這份 spec 的目的是把專案收斂成「老師容易理解、流程容易驗收、功能重點明確」的作業版本。

## Goal
- 將專案整理成一個更像正常作業專案的花店購物網站。
- 使用者流程要清楚完成：瀏覽商品、加入購物車、會員登入、登入後結帳、綠界付款、付款回站查單。
- 提供老師可直接使用的 Admin 帳號密碼展示區。
- 提供清楚的測試文件，讓老師可依步驟驗收。

## Non-Goals
- 不追求完整正式商業電商功能。
- 不做真實會員註冊、忘記密碼寄信、完整權限系統。
- 不做真實訂單管理後台、物流、發票、優惠券。
- 不額外擴充與作業需求無直接關聯的品牌故事或過度包裝頁面。

## Audience
- 老師 / reviewer：需要快速看懂作業功能與驗收流程。
- 一般使用者：能完成從購物到付款的基本流程。
- 開發者 / AI Agent：能依 spec 收斂 UI 與流程，不再持續擴 scope。

## Product Positioning
- 專案定位為：
  - 單一品牌花店電商作業網站
  - 具備基本購物流程
  - 具備會員登入後才能結帳的門檻
  - 具備綠界測試付款流程
  - 具備老師可驗收的 Admin 示意入口

## User Stories
### Shopper
- 作為訪客，我想瀏覽商品與加入購物車。
- 作為訪客，我在前往結帳時會被要求先登入會員。
- 作為會員，我登入後可以進入結帳頁並完成付款流程。
- 作為會員，我希望付款後能回到網站並看到付款結果。

### Reviewer / Teacher
- 作為老師，我希望能快速看到測試用 Admin 帳號密碼。
- 作為老師，我希望有一份文件可以照著測完整流程。
- 作為老師，我希望能在本地端驗證付款結果，而不是只看到前端假畫面。

## Functional Requirements

### 1. 商品瀏覽
- 提供商品列表頁。
- 商品需包含：
  - 圖片
  - 商品名稱
  - 分類
  - 原價
  - 會員價
  - 商品描述
- 必備商品類別至少包含：
  - 蝴蝶蘭
  - 花籃
- 可保留其他類別，但不得取代上述必備類別。

### 2. 商品詳情
- 使用者可查看單一商品詳細資訊。
- 可選擇數量並加入購物車。
- 原價與會員價需清楚區分，會員價視覺上需更明顯。

### 3. 購物車
- 使用者可在購物車查看已選商品。
- 可調整數量、刪除商品。
- 可看到總金額。
- 若未登入會員，仍可加入購物車，但不可直接完成結帳。

### 4. 會員登入限制
- 使用者未登入時：
  - 可以瀏覽商品
  - 可以加入購物車
  - 可以查看購物車
  - 不可以進入結帳頁
- 嘗試前往 `/checkout` 時，系統需明確提示先登入會員。
- 登入成功後，才允許進入結帳頁。
- 會員登入可使用 mock 流程，但畫面與文案要簡潔，不要過度包裝。

### 5. 結帳流程
- 只有會員可進入結帳頁。
- 結帳頁需包含：
  - 收件人姓名
  - Email
  - 電話
  - 配送資訊
  - 訂單摘要
  - 付款方式（綠界）
- 送出後建立訂單並導向綠界測試付款頁。

### 6. 綠界金流
- 使用綠界測試環境。
- 需由本地 server 建立交易資料與簽章。
- 付款完成後需回到前端結果頁。
- 因本地端通常無法直接依賴 ReturnURL 作為成功依據，所以需：
  - 以前端取得 `merchantTradeNo`
  - 呼叫本地 server 查詢綠界訂單結果
  - 以查單結果顯示付款成功 / 失敗 / 未完成

### 7. 付款結果頁
- 需存在獨立付款結果頁，例如 `/payment-result`。
- 頁面至少顯示：
  - 訂單編號 / MerchantTradeNo
  - 付款狀態
  - 付款方式
  - 付款時間（若查得到）
  - 原始查單摘要或可讀訊息
- 若查詢失敗，需顯示可理解的錯誤訊息。

### 8. Admin 驗收區
- 提供明確 Admin 入口。
- Admin 登入頁需直接展示測試帳號密碼，方便老師驗收。
- 建議至少顯示：
  - 測試帳號
  - 測試密碼
  - 此區用途說明
- Admin 可為示意後台，但至少要能讓老師看到：
  - 商品管理區
  - 或商品列表 / 編輯能力

### 9. 測試文件
- 專案需提供可直接照做的測試文件。
- 建議放在 `docs/TESTING.md`。
- 測試文件需包含：
  - 安裝與啟動步驟
  - 環境變數說明
  - 前台購物流程
  - 會員登入流程
  - 結帳與綠界付款流程
  - 付款結果確認方式
  - Admin 登入方式

## UX / Flow

### Primary User Flow
1. 進入首頁或商品列表頁。
2. 瀏覽商品。
3. 查看商品詳情。
4. 加入購物車。
5. 進入購物車。
6. 點擊前往結帳。
7. 若未登入，跳出提示並導向會員登入。
8. 登入成功後回到結帳流程。
9. 填寫收件資料與確認訂單。
10. 送出並導向綠界測試付款頁。
11. 完成付款後返回前端付款結果頁。
12. 前端再透過本地 server 查詢綠界結果並顯示。

### Teacher Review Flow
1. 啟動專案。
2. 依 `docs/TESTING.md` 進行前台購物測試。
3. 使用測試會員登入。
4. 完成綠界付款測試。
5. 返回付款結果頁確認查單結果。
6. 進入 Admin 頁面。
7. 使用畫面提供的帳密登入。
8. 查看後台示意功能。

## Content / Tone Direction
- 專案整體文案應收斂成「作業可驗收」導向。
- 避免過度像 AI 生成的症狀：
  - 過長品牌故事
  - 過多華麗但非必要的敘事區塊
  - 過多重複的提示字卡或 showcase 文案
  - 與作業主流程無關的複雜互動
- 優先保留：
  - 清楚功能
  - 乾淨畫面
  - 易懂動線
  - 老師可快速驗收的資訊

## UI Guidelines
- 風格保持花店品牌感，但不要蓋過功能本身。
- 首頁可簡潔展示品牌與主打品類即可。
- 會員 modal / 會員頁文案應簡化。
- Admin 登入頁需比現在更像驗收入口，而不是一般商業後台登入頁。
- 會員價必須比原價更突出。
- 桌機與手機版都需可用。

## Edge Cases
- 未登入直接輸入 `/checkout`。
- 購物車為空仍試圖進結帳。
- 綠界返回後缺少 `merchantTradeNo`。
- 查單失敗或綠界查無結果。
- 本地 server 未啟動時，前端應顯示可理解訊息。
- Admin 帳密輸入錯誤時，需顯示清楚錯誤。

## Non-Functional Requirements
- 專案結構需維持清楚，可讓 AI Agent 與人類都看懂。
- `AGENTS.md` / `CLAUDE.md` 維持精簡。
- `docs` 內文件應對應各自主題，不混亂堆疊。
- 功能完成後需可通過至少：
  - build
  - lint（若專案有設定）
- 不新增不必要的大型依賴。

## Acceptance Criteria
- 使用者可成功瀏覽商品並加入購物車。
- 未登入使用者無法直接進入結帳頁。
- 登入會員後可進入結帳頁。
- 可從前端建立綠界測試訂單並導向付款頁。
- 完成付款後可回到前端付款結果頁。
- 前端可透過本地 server 查單並顯示付款狀態。
- Admin 頁面可明確看到測試帳號密碼。
- `docs/TESTING.md` 可讓老師依步驟完成驗收。
- 專案整體文案與結構比目前更收斂，不再像 AI showcase。

## Deliverables
- 一份收斂後的前台購物流程。
- 一個有登入門檻的結帳流程。
- 一個可顯示付款結果的前端頁面。
- 一個方便老師驗收的 Admin 登入入口。
- 一份更新後的測試說明文件。

## Open Questions
- 測試會員帳號密碼是否需要固定展示在前端某處，或只放在測試文件中？
- Admin 後台是否只需商品管理，或還要保留會員資料列表？
- 是否保留目前較長的 About / 品牌敘事頁，還是收斂成簡版？
- 付款結果頁要顯示多少技術細節，才能兼顧可讀性與驗收需求？

## Assumptions
- 目前既有綠界 server 流程可延用，只需補齊前端付款結果頁與驗收動線。
- 會員登入可維持 mock，不需實作真實註冊與忘記密碼流程。
- Admin 為驗收用途，可接受使用固定測試帳號密碼。
- 本 spec 以作業驗收優先，不以完整商業網站為優先。
